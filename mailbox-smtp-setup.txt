mailbox.org SMTP Setup — gehaltsdeckel.jetzt
=============================================

GOAL: Send transactional emails FROM noreply@gehaltsdeckel.jetzt via a
      personal mailbox.org account. Cloudflare Email Routing (forwarding
      to Gmail) remains active and is unaffected by this setup.

SMTP SERVER
-----------
  Host:     smtp.mailbox.org
  Port:     587
  Security: STARTTLS (secure: false in nodemailer — no code change needed)
  Auth:     your full mailbox.org address + account password
            (no app password needed — standard credentials work)


STEP 1 — Add noreply@gehaltsdeckel.jetzt as a sender alias in mailbox.org
--------------------------------------------------------------------------
mailbox.org must verify you own the domain before allowing you to send from it.

1a. Log in to mailbox.org → gear icon (top right) → All settings
    → Email addresses → Email aliases section
    → "Add external alias" → enter: noreply@gehaltsdeckel.jetzt → Save

1b. mailbox.org will show a validation error containing a security key in
    two parts, e.g.:
      Part 1 (hostname): p30oe995fc9105322345b6277ac3b1i2eb45106j5j7
      Part 2 (value):    IN TXT 5ahgui6nsdj5kkk666l333m7m7k7n5k6nnnn5bb4

1c. In Cloudflare DNS for gehaltsdeckel.jetzt, add a TXT record:
      Type:    TXT
      Name:    p30oe995fc9105322345b6277ac3b1i2eb45106j5j7   ← use your actual Part 1
      Content: 5ahgui6nsdj5kkk666l333m7m7k7n5k6nnnn5bb4     ← use your actual Part 2
      TTL:     Auto

1d. DNS propagation can take up to 48 hours. Once complete, mailbox.org
    confirms the alias and noreply@gehaltsdeckel.jetzt becomes a usable
    sender identity on your account.


STEP 2 — DNS records for SPF, DKIM, and DMARC
----------------------------------------------
Add these records in Cloudflare DNS for gehaltsdeckel.jetzt.
In Cloudflare, proxy status for all of these must be DNS only (grey cloud).

  SPF (merge with any existing SPF record — only ONE TXT on @ is allowed):
    Type:    TXT
    Name:    @
    Content: v=spf1 include:_spf.mx.cloudflare.net include:mailbox.org ~all
    Note:    include:_spf.mx.cloudflare.net covers Cloudflare Email Routing
             (forwarding to Gmail). If that include is already present, just
             add include:mailbox.org to the existing record.

  DKIM (four CNAME records — mailbox.org rotates keys automatically via CNAME):
    Type:    CNAME   Name: MBO0001._domainkey   Target: MBO0001._domainkey.mailbox.org.
    Type:    CNAME   Name: MBO0002._domainkey   Target: MBO0002._domainkey.mailbox.org.
    Type:    CNAME   Name: MBO0003._domainkey   Target: MBO0003._domainkey.mailbox.org.
    Type:    CNAME   Name: MBO0004._domainkey   Target: MBO0004._domainkey.mailbox.org.
    Note:    Cloudflare may or may not require a trailing dot on the target —
             add it if Cloudflare doesn't append it automatically.

  DMARC:
    Type:    TXT
    Name:    _dmarc
    Content: v=DMARC1;p=none;rua=mailto:postmaster@gehaltsdeckel.jetzt
    TTL:     400
    Note:    p=none means no action on failures — only reporting. Keep this
             value until you have confirmed SPF and DKIM are working, then
             optionally tighten to p=quarantine or p=reject.
             Always include rua= — some receiving servers reject mail if it
             is missing even though the RFC makes it optional.

  MX records: DO NOT CHANGE. Cloudflare Email Routing MX records stay as-is.
              They handle inbound/forwarding to Gmail and are independent of
              outbound authentication.


STEP 3 — Add credentials to .env on the server
------------------------------------------------
  MAILBOX_USER=your@mailbox.org
  MAILBOX_PASSWORD=your-mailbox-account-password


STEP 4 — Verify DNS records (macOS / Linux)
--------------------------------------------
  # SPF
  dig gehaltsdeckel.jetzt TXT | grep spf

  # DKIM (check one of the four CNAMEs — all should resolve)
  dig MBO0001._domainkey.gehaltsdeckel.jetzt CNAME

  # DMARC
  dig _dmarc.gehaltsdeckel.jetzt TXT

  Each should return a non-empty ANSWER SECTION. If empty, DNS has not
  propagated yet — wait and retry.


STEP 5 — Deploy
----------------
On the server:

  docker compose up -d --build

Haraka is not part of the production mail path. The app connects directly to
smtp.mailbox.org:587 with MAILBOX_USER and MAILBOX_PASSWORD.


STEP 6 — Verify
-----------------
Watch app logs for nodemailer errors:

  docker compose logs -f app

A successful send looks like:
  [email] sent via=smtp.mailbox.org:587 auth=yes toDomain=example.org messageId=<...> response="250 ..."

If the app exits with "Production email requires SMTP_USER and SMTP_PASS",
check MAILBOX_USER and MAILBOX_PASSWORD in .env or your deployment settings.

If smtp.mailbox.org rejects authentication, confirm that MAILBOX_USER is the
full mailbox.org account address and MAILBOX_PASSWORD is the matching account
password.

If a mailbox.org-hosted recipient later rejects or delays delivery, collect the
bounce SMTP status and diagnostic text. Do not reintroduce Haraka for that case;
first verify the sender alias, SPF, DKIM, and DMARC alignment above.


NOTES
-----
- mailbox.org handles outbound DKIM signing when the sender alias and DKIM
  CNAME records are configured correctly.
- Cloudflare Email Routing remains inbound-only and is independent of this
  outbound SMTP setup.
- The Haraka files in this repo are legacy fallback material only. They are not
  used by docker-compose.yml.
